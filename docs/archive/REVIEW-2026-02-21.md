## 1. Current State Summary

VoicePaste has grown from a simple voice-to-text MVP (v0.1) into a feature-rich desktop tool with 25 source modules and ~5900 statements. Key capabilities:

- Cloud + local STT (OpenAI Whisper / faster-whisper)
- Cloud + local TTS (ElevenLabs / Piper)
- Three summarization providers (OpenAI, OpenRouter, Ollama) + passthrough
- Voice Prompt Q&A mode
- Hands-free wake-word activation
- HTTP API for external control
- Confirm-before-paste with preview
- Cross-platform support (Windows + Linux)
- Settings hot-reload without restart

**Core strength**: The tool "just works" for the single-recording flow. Clean protocol-based backends, thread-safe state machine, robust error handling, and strong security posture.

**Core weakness**: The pipeline is strictly serial -- all hotkey presses during PROCESSING are silently ignored. This is the biggest UX friction point for power users.

## 2. User-Proposed Features -- Evaluation

### 2.1 Multiple Voice Output (Simulating Multiple People)

**Verdict: DEFERRED**

| Perspective | Assessment |
|-------------|------------|
| **Product** | Secondary feature of a secondary feature. Doesn't serve the core dictation use case. |
| **Architecture** | ElevenLabs: easy (per-request voice switching). Piper: hard (multiple ~100MB ONNX models in RAM, needs a `VoicePool`). |
| **UX** | No natural trigger point. Would require text annotation with speaker roles, conflicting with "invisible, zero-friction" philosophy. |
| **Security** | Voice cloning models = biometric data requiring GDPR Art. 9 protections. |

**If revisited later**: A simpler "Quick Voice Switch" (tray submenu with voice presets) is the recommended first step. Full multi-speaker synthesis is v2.0+ territory.

#### Quick Voice Switch Design (for future reference)
- Tray right-click menu: "TTS Voice >" submenu with radio-dot for active voice
- Max 5 presets (prevents menu bloat)
- Each preset stores: name, provider, voice_id/voice_name
- Settings dialog: "Voice Presets" section in TTS tab
- Config: new `[[tts.presets]]` section in config.toml

### 2.2 Pipeline Queueing (Record During Processing)

**Verdict: ACCEPTED -- Priority #2**

This directly eliminates the biggest UX friction point. Currently in `main.py:1143`:
```python
elif current == AppState.PROCESSING:
    logger.info("Hotkey pressed during PROCESSING state, ignored.")
```

Users must wait 3-8 seconds (cloud) or 2-5 seconds (local) before starting the next recording.

#### Approved Design

| Aspect | Decision |
|--------|----------|
| Queue depth | 1 (max 2 in flight: 1 processing + 1 recording) |
| Tray icon | Split-color: left/top RED (recording), right/bottom AMBER (processing) |
| State model | No new AppState enum -- use `_queued_audio: Optional[bytes]` flag |
| Paste ordering | FIFO (first recorded = first pasted) |
| Clipboard | Backup before first item, restore after last queued item pastes |
| Cancel (Escape) | During queued recording: cancels only the queued recording |
| Queue full cue | 660 Hz, 50ms busy tone |
| Scope exclusions | Not supported with TTS Ask mode or AWAITING_PASTE |
| Setting | Always on (pure UX improvement, no reason to disable) |

#### State Machine Changes (Phase 1)
```
Current:  IDLE -> RECORDING -> PROCESSING -> PASTING -> IDLE
                                           -> AWAITING_PASTE -> PASTING -> IDLE

Phase 1:  PROCESSING + hotkey -> start recording (queued)
          PROCESSING + hotkey again -> stop queued recording, store audio
          Pipeline completes -> process queued audio -> paste -> IDLE
```

#### Architecture Approach
- Add `_queued_audio: Optional[bytes]` to VoicePasteApp
- In `_on_hotkey()`: allow starting recording when state is PROCESSING
- In `_run_pipeline()` finally block: check for queued audio, process it before returning to IDLE
- Clipboard backup taken once (first recording), restored after last paste

| Audio Data Protection | **A** | In-memory only, buffer zeroing, never logged |

## 6. UX Observations

### Current Pain Points
1. **Silent rejection during PROCESSING** -- user presses hotkey, nothing happens, no feedback
2. **No progress indication** beyond tray icon color change during processing
3. **Language switching** requires opening Settings dialog

### UX Design Documents Created
- `pipeline-queueing-design.md` -- Split-color tray icon, queue-full cue, cancel behavior
- `multi-voice-design.md` -- Quick Voice Switch via tray submenu (deferred)

### Key UX Principles Confirmed
- The tool should be invisible and non-intrusive
- Audio cues > visual notifications (users don't stare at the tray)
- Distinct cues for distinct states: queued != rejected != started
- Static tray icon states preferred over animation (no animation infrastructure)

## 7. Architecture Debt Summary

### main.py God Object (1925 lines)
The `VoicePasteApp` class is simultaneously the state machine, pipeline executor, API dispatcher, TTS orchestrator, and settings coordinator. Recommended extraction order:
1. `APIController` (dispatch logic, ~200 lines)
2. `TTSOrchestrator` (synthesis, cache, export, playback, ~300 lines)
3. Keep `VoicePasteApp` as thin orchestrator

### Implicit State Machine
State transitions spread across 15+ methods. No single place shows all valid transitions. Recommendation: add a `_transition_state(from, to)` method with an explicit transition table dict.

### HotkeyManager Duplication
Same 40-line pattern repeated 5 times. Recommendation: `_HotkeySlot` dataclass with generic `_register_slot()` method.
